# 역할 및 목표
당신은 개발자용 노트 앱(Notion 유사) 분야의 시니어 기획자입니다.
아래 기능에 대한 기획 문서를 작성해 주세요. 이 문서는 이후 Claude를 통한
코드 구현 단계에서 그대로 입력으로 사용될 예정이므로, 구현에 필요한 수준의
구체성을 갖춰야 합니다.

# 기능 개요
- 기능명: 노트 아카이빙 (Note Archive)
- 한 줄 설명: 완료되었거나 더 이상 활발히 쓰지 않는 노트를 삭제하지 않고 메인 목록에서 분리해 보관하는 기능
- 배경/문제 상황: 노트가 누적되면서 메인 목록(Home)과 보드(Board)가 오래된 노트로 혼잡해짐. 삭제(휴지통)는 30일 후 영구 삭제되므로 장기 보관 용도로 부적합. 참고용으로 영구 보존하되 일상 작업 화면에서는 치워두는 중간 상태가 필요함.

# 기술 컨텍스트
- 언어/프레임워크: C# / .NET 10
  - 백엔드: `Vanadium.Note.REST` — ASP.NET Core Web API, EF Core + PostgreSQL, JWT 인증
  - 프론트엔드: `Vanadium.Note.Web` — Blazor WebAssembly, MudBlazor, Tiptap 에디터(JS interop)
- 관련 시스템/모듈:
  - `NoteItem` 엔티티 및 `NoteService` (소프트 삭제용 글로벌 쿼리 필터 `DeletedAt == null` 존재)
  - 휴지통 기능: `DeletedAt`/`IsDeletionRoot` 기반 소프트 삭제, `RecycleBinPurgeJob`, 복원/영구삭제 API — 아카이브는 이와 유사하지만 독립적인 패턴으로 설계할 것
  - 전문 검색: `Title` + `ContentText`의 PostgreSQL `tsvector` 컬럼 + GIN 인덱스 (`ILIKE` 검색 금지)
  - `OrphanFileCleanupJob` / `FileCleanupService`: 노트 HTML 내 `/uploads/file_{guid}` 참조를 스캔해 고아 파일 삭제 — 아카이브된 노트의 첨부 파일은 살아있는 참조로 취급되어야 함
  - 프론트 페이지: `/` (Home, 노트 목록), `/board` (칸반), `/editor/{id}`, `/recycle-bin`
- 외부 표준/의존성: 신규 외부 의존성 추가 금지 (의존성 표면 최소화 원칙)
- 제약 조건:
  - 스키마 변경 시 EF Core 마이그레이션 필수, 기존 마이그레이션 파일 수정 금지
  - DTO는 REST/Web 프로젝트에 의도적으로 중복 정의 — 공유 프로젝트 도입 금지
  - 코드/주석/UI 텍스트는 전부 영어
  - 노트 콘텐츠를 스캔하거나 일괄 삭제하는 신규 코드는 `IgnoreQueryFilters()` 필요 여부를 반드시 검토할 것 (아카이브에 글로벌 쿼리 필터를 추가할 경우 기존 휴지통·파일 정리·계정 삭제 경로에 미치는 영향 포함)

# 확정된 요구사항 (기획 시 반드시 반영)
1. **하위 노트 동반 아카이브**: 노트를 아카이브하면 활성 하위 노트도 함께 아카이브된다. 복원 시 함께 복원된다. (휴지통의 `IsDeletionRoot` 패턴과 유사한 루트 추적 방식 검토)
2. **읽기 전용**: 아카이브된 노트는 열람 가능하지만 제목/내용/라벨 수정이 불가하다. 서버 측에서 강제하고, 프론트는 에디터를 읽기 전용 모드로 표시한다.
3. **노출 정책**: 메인 목록(Home)과 보드(Board)에서는 제외. 전문 검색 결과에는 포함하되, 결과 항목에 "Archived" 표시를 붙인다. 별도 아카이브 페이지(예: `/archive`)에서 목록 조회·복원 가능.
4. **휴지통과의 상호작용 정의 필요**: 아카이브된 노트의 삭제 가능 여부, 휴지통 복원 시 아카이브 상태 처리 등 상태 전이를 명확히 정의할 것.
5. **자동 영구 삭제 없음**: 휴지통과 달리 아카이브는 무기한 보존이다.
