# 역할 및 목표
당신은 개발자용 노트 앱(Notion 유사) 분야의 시니어 기획자입니다.
아래 기능에 대한 기획 문서를 작성해 주세요. 이 문서는 이후 Claude를 통한
코드 구현 단계에서 그대로 입력으로 사용될 예정이므로, 구현에 필요한 수준의
구체성을 갖춰야 합니다.

# 기능 개요
- 기능명: 빠른 탐색 (Quick Navigation)
- 한 줄 설명: 단축키로 띄우는 명령 팔레트(Quick Switcher)에서 노트를 검색·선택해 즉시 이동하고, 기존 전문 검색 경험을 키보드 중심의 빠른 탐색 UX로 강화하는 기능
- 배경/문제 상황: 노트가 누적되면 원하는 노트로 이동하는 데 Home 목록 스크롤이나 검색 페이지 왕복이 필요해 흐름이 끊긴다. 마우스 없이 키보드만으로 어느 화면에서든 즉시 노트를 찾아 이동할 수 있는 진입점이 없다. Notion의 Quick Find, VSCode의 Ctrl+P처럼 단축키 → 입력 → 선택 → 이동의 빠른 경로가 필요하다.

# 기술 컨텍스트
- 언어/프레임워크: C# / .NET 10
  - 백엔드: `Vanadium.Note.REST` — ASP.NET Core Web API, EF Core + PostgreSQL, JWT 인증
  - 프론트엔드: `Vanadium.Note.Web` — Blazor WebAssembly, MudBlazor, Tiptap 에디터(JS interop)
- 관련 시스템/모듈:
  - `NoteItem` 엔티티 및 `NoteService` — 노트 조회/검색의 핵심
  - 전문 검색: `NoteItem.Title` + `ContentText`에 PostgreSQL GIN **트라이그램** 인덱스(`gin_trgm_ops`). `NoteService.ApplyFilters()`가 공백 단위 term마다 `EF.Functions.ILike`를 적용하며 트라이그램 인덱스로 가속됨. 검색 결과는 아카이브 노트를 포함하며 `NoteSummary.IsArchived` 플래그로 표시. **빠른 탐색의 노트 검색은 이 기존 검색 경로를 재사용/확장할 것 — 인덱스되지 않은 컬럼에 `ILIKE` 금지**
  - 아카이브: `ArchivedAt` 기반. 전문 검색에는 포함되나 Home/Board/children/mention 검색에서는 제외(`Where(n => n.ArchivedAt == null)`). 빠른 탐색 결과에는 아카이브 노트를 **포함**하고 "Archived" 배지로 구분
  - 소프트 삭제: `DeletedAt == null` 글로벌 쿼리 필터가 `NoteItem`에 적용됨 — 휴지통 노트는 빠른 탐색 결과에서 제외되어야 함(기본 필터로 자동 제외됨)
  - 인증/사용자 컨텍스트: 모든 노트 조회는 현재 사용자 범위로 한정. JWT는 `TokenStore` + `AuthTokenHandler`로 주입됨
  - 프론트 페이지/라우팅: `/` (Home), `/board`, `/editor/{id}`, `/archive`, `/recycle-bin`. Blazor `NavigationManager`로 이동
  - 전역 단축키: 현재 앱 전역 키보드 단축키 인프라가 없으므로, 어느 페이지에서든 동작하는 전역 단축키 처리 방식(예: `MainLayout` 수준의 JS interop `keydown` 리스너 또는 MudBlazor `MudPopover`/`MudOverlay` 기반 다이얼로그)을 설계에 포함할 것
- 외부 표준/의존성: 신규 외부 의존성 추가 금지 (의존성 표면 최소화 원칙). MudBlazor 기본 컴포넌트와 기존 JS interop 패턴 활용
- 제약 조건:
  - 스키마 변경 시 EF Core 마이그레이션 필수, 기존 마이그레이션 파일 수정 금지
  - DTO는 REST/Web 프로젝트에 의도적으로 중복 정의 — 공유 프로젝트 도입 금지
  - 신규 API 엔드포인트는 (1) REST DTO, (2) Web DTO 미러, (3) `NoteService` HTTP 클라이언트 메서드를 모두 추가
  - 코드/주석/UI 텍스트는 전부 영어
  - 노트 콘텐츠를 스캔하거나 일괄 조회하는 신규 코드는 `IgnoreQueryFilters()` 필요 여부를 검토할 것 (빠른 탐색은 활성+아카이브만 대상, 휴지통 제외이므로 기본 글로벌 필터를 그대로 사용하면 됨 — 명시할 것)

# 확정된 요구사항 (기획 시 반드시 반영)
1. **명령 팔레트 + 검색 강화 동시 제공**: 전역 단축키(예: `Ctrl/Cmd+K`)로 어느 화면에서든 빠른 탐색 다이얼로그를 띄운다. 동시에 기존 전문 검색 경험을 이 빠른 탐색 UX(실시간 결과, 키보드 탐색)로 강화한다. 둘의 관계(별도 진입점인지, 검색 페이지가 동일 컴포넌트를 재사용하는지)를 정의할 것.
2. **탐색 대상**:
   - **노트 제목/내용**: 기존 트라이그램 전문 검색(`Title` + `ContentText`)을 재사용한다.
   - **최근 방문 노트(Recents)**: 입력이 비어 있을 때 최근 열어본 노트를 우선 노출한다. 최근 방문 기록의 저장 위치(클라이언트 `localStorage` vs 서버 추적)와 보존 개수를 설계에서 결정할 것. 신규 스키마/엔드포인트가 필요한지 트레이드오프와 함께 제시.
   - **아카이브 포함**: 아카이브된 노트도 결과에 포함하고 "Archived" 배지로 표시한다. (휴지통 노트는 제외)
   - (이번 범위에서 **명령/액션 실행은 제외** — 순수 노트 탐색·이동에 집중)
3. **키보드 우선 인터랙션**: 입력 즉시 실시간 결과 갱신, `↑`/`↓`로 항목 이동, `Enter`로 선택 노트로 이동, `Esc`로 닫기. 마우스 클릭도 지원하되 키보드만으로 전 과정 완결 가능해야 함.
4. **성능/응답성**: 입력에 디바운스를 적용해 과도한 요청을 막고(기존 에디터 auto-save의 디바운스 패턴 참고 가능), 결과 개수 상한(예: 상위 N개)을 둔다. 트라이그램 인덱스를 활용하는 쿼리 형태를 유지할 것.
5. **결과 표시**: 각 결과 항목에 제목과 본문 일부 미리보기(스니펫), 아카이브 배지를 포함. 매칭 위치 하이라이트 여부는 기획에서 결정.

# 작성해야 할 항목
1. 요구사항 정의 (기능/비기능 요구사항을 구분)
2. 사용자 시나리오 / 유스케이스 (예: 빈 입력 시 최근 노트 → 입력 시 검색, 아카이브 노트로 이동, 검색 페이지에서의 동작)
3. 입력·출력 명세 (API 엔드포인트, DTO 구조, 타입 포함 — 기존 검색 엔드포인트 재사용 여부와 최근 방문 노트 조회 방식 포함)
4. 처리 흐름 (단계별 로직, 필요 시 의사코드 — 특히 디바운스된 입력 처리, 빈 입력 시 Recents 노출, 검색 쿼리 구성, 결과 랭킹/정렬)
5. 인터페이스 설계 (프론트 컴포넌트 구조 및 전역 단축키 처리, 필요한 엔티티/DTO 변경, 클래스/메서드 시그니처 초안, 필요 시 EF Core 마이그레이션 개요)
6. 예외 및 엣지 케이스 처리 (예: 결과 없음, 매우 짧은 입력, 권한 없는/삭제된 노트로의 이동 시도, 최근 방문 기록이 가리키는 노트가 휴지통/영구삭제된 경우, 단축키 충돌)
7. 테스트 시나리오 (정상/경계/실패 케이스 — 백엔드 검색 로직은 신규/기존 `Vanadium.Note.REST.Tests`에서 커버하되, 트라이그램 `ILike`는 PostgreSQL 전용이라 수동 검증 영역임을 명시)
8. 구현 시 고려사항 / 미결정 사항(open questions — 예: Recents 저장 위치, 결과 랭킹 기준, 검색 페이지 통합 범위)
