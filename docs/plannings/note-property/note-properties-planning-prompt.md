# 역할 및 목표
당신은 개발자용 노트 앱(Notion 유사) 분야의 시니어 기획자입니다.
아래 기능에 대한 기획 문서를 작성해 주세요. 이 문서는 이후 Claude를 통한
코드 구현 단계에서 그대로 입력으로 사용될 예정이므로, 구현에 필요한 수준의
구체성을 갖춰야 합니다.

# 기능 개요
- 기능명: 노트 프로퍼티 (Note Properties)
- 한 줄 설명: 노션의 프로퍼티처럼 노트에 구조화된 메타데이터(N개의 타입 있는 속성 값)를 부여하고, 그 값으로 노트 목록을 필터·정렬할 수 있게 하는 기능
- 배경/문제 상황: 현재 노트의 구조화된 메타데이터는 Label(카테고리 내 상호배타 라벨)뿐이라 표현력이 부족하다. "마감일", "우선순위 숫자", "완료 여부", "진행 상태" 같은 타입 있는 속성을 노트에 붙일 수 없고, 이런 속성 기준으로 Home/Board 목록을 필터·정렬할 수도 없다. 노션의 프로퍼티처럼 노트마다 여러 속성 값을 부여하고 목록에서 활용할 수 있는 구조화 계층이 필요하다.

# 기술 컨텍스트
- 언어/프레임워크: C# / .NET 10
  - 백엔드: `Vanadium.Note.REST` — ASP.NET Core Web API, EF Core + PostgreSQL, JWT 인증(단일 사용자, 소유권 컬럼 없음)
  - 프론트엔드: `Vanadium.Note.Web` — Blazor WebAssembly, MudBlazor, Tiptap 에디터(JS interop)
- 관련 시스템/모듈:
  - `NoteItem` 엔티티 및 `NoteService` — 노트 CRUD/조회의 핵심. `PUT /api/notes/{id}`는 낙관적 동시성 검사를 수행하며, 아카이브된 노트는 읽기 전용(수정 403)
  - Label 시스템: `Label` + 선택적 `LabelCategory`, 조인 테이블 `NoteLabel`(복합 PK `(NoteId, LabelId)`, `DeletedAt` 매칭 글로벌 필터 있음). 카테고리 내 상호배타는 DB 제약이 아니라 `NoteService`에서 서버측 강제. `NoteItem.Labels`는 `[NotMapped]`이며 `PopulateLabels()`로 채워짐 — **Label은 사실상 단일 Select 프로퍼티와 동형이므로 이번 기획의 통합 검토 대상**
  - 소프트 삭제/휴지통: `NoteItem`에 `DeletedAt == null` 글로벌 쿼리 필터. 휴지통 관련 경로만 `IgnoreQueryFilters()` 사용. **노트를 일괄 조회·스캔하거나 연쇄 삭제하는 신규 코드는 `IgnoreQueryFilters()` 필요 여부를 반드시 검토할 것** (예: 프로퍼티 정의 삭제 시 소프트 삭제된 노트의 값도 함께 정리해야 복원이 일관됨)
  - 아카이브: `ArchivedAt` 기반, 글로벌 필터 없이 명시적 `Where` 사용. 아카이브 노트는 읽기 전용이므로 프로퍼티 값 편집도 차단되어야 함
  - 전문 검색: `Title` + `ContentText`에 PostgreSQL GIN 트라이그램 인덱스. 프로퍼티 값의 검색 포함 여부는 기획에서 결정하되, **인덱스되지 않은 컬럼에 `ILIKE` 금지** 원칙 준수
  - 프론트 페이지: `/` (Home, 노트 목록 + 검색/필터), `/board` (칸반 보드), `/editor/{id}` (Tiptap 에디터), `/archive`, `/recycle-bin`. 프로퍼티 편집 UI는 에디터에, 필터·정렬 UI는 Home/Board에 배치될 것으로 예상
  - 계정 초기화: `AccountService`의 전체 삭제 경로가 신규 테이블도 함께 지워야 함
- 외부 표준/의존성: 신규 외부 의존성 추가 금지 (의존성 표면 최소화 원칙). MudBlazor 기본 컴포넌트 활용
- 제약 조건:
  - 스키마 변경 시 EF Core 마이그레이션 필수, 기존 마이그레이션 파일 수정 금지
  - DTO는 REST/Web 프로젝트에 의도적으로 중복 정의 — 공유 프로젝트 도입 금지
  - 신규 API 엔드포인트는 (1) REST DTO, (2) Web DTO 미러, (3) Web 측 HTTP 클라이언트 메서드를 모두 추가하고, `docs/api-specification.md`를 같은 변경에서 갱신
  - 코드/주석/UI 텍스트는 전부 영어
  - 단일 사용자 앱 — `UserId` 같은 소유권 컬럼을 새 테이블에 추가하지 말 것

# 확정된 요구사항 (기획 시 반드시 반영)
1. **전역 프로퍼티 정의 (global definitions)**: 프로퍼티의 이름·타입·(Select 계열이면) 옵션 목록을 전역으로 정의하고, 노트에는 값만 부여한다. 노트별 자유 정의(ad-hoc)나 혼합형(노트별 선택 부착)은 채택하지 않는다 — 이 결정의 근거(필터·정렬과의 상성, 옵션·이름의 단일 관리)는 문서에 간단히 기록할 것. 값이 없는 프로퍼티는 해당 노트에서 빈 값으로 취급하며, 빈 값 프로퍼티를 에디터에 항상 표시할지/숨길지는 기획에서 결정한다. 전역 정의를 관리(생성/이름 변경/타입 변경/옵션 관리/삭제)하는 UI 동선도 설계에 포함할 것.
2. **1차 지원 타입 (기본 세트)**: `Text`, `Number`, `Select`, `MultiSelect`, `Date`, `Checkbox`의 6종. 이후 확장 후보(URL, Status, 노트 Relation 등)는 open questions에만 기록하고 1차 범위에서 제외한다. 값 저장 모델(타입별 컬럼을 가진 값 테이블(EAV) vs JSONB 컬럼 vs 그 혼합)은 기획에서 **필터·정렬 쿼리 형태와 인덱스 전략을 근거로** 비교·결정할 것.
3. **Label 시스템 통합/마이그레이션 검토**: Label은 Select 프로퍼티와 개념적으로 동형이다(LabelCategory ≈ Select 정의, 카테고리 내 상호배타 ≈ 단일 선택, 무카테고리 라벨 ≈ MultiSelect 옵션에 근접). 기획에서 (a) Label을 프로퍼티로 흡수하는 마이그레이션 방안(데이터 마이그레이션 절차, 기존 Label API/UI의 거취 포함)과 (b) 병행 유지 방안을 비교하고 **권고안을 결정**할 것. 흡수를 권고하는 경우 1차 범위에 포함할지, 프로퍼티 출시 후 후속 단계로 뺄지도 명시한다.
4. **필터·정렬 포함 (1차 범위)**: Home 목록에서 프로퍼티 값 기반 필터(예: Checkbox=checked, Select=특정 옵션, Date 범위)와 정렬(예: Number 오름차순, Date 내림차순)을 제공한다. Board에서의 활용 범위(예: Select 프로퍼티를 보드 컬럼 기준으로 쓸지, 카드에 값 배지만 표시할지)는 기획에서 결정한다. 필터·정렬 조건의 API 표현(쿼리 파라미터 설계)과 서버측 쿼리 구성, 필요한 인덱스를 구체적으로 설계할 것.
5. **에디터 내 프로퍼티 패널**: 노션처럼 에디터의 제목 아래·본문 위 영역에 프로퍼티 패널을 두고, 값 편집은 타입별 적절한 입력 컨트롤(MudBlazor)로 제공한다. 프로퍼티 값 변경은 기존 노트 auto-save(1500ms 디바운스)와 어떻게 상호작용할지(같은 저장 요청에 실리는지, 별도 즉시 저장인지) 기획에서 결정할 것. 아카이브된 노트에서는 프로퍼티도 읽기 전용이다. Tiptap 본문 콘텐츠와는 무관한 기능이므로 에디터 JS interop 변경은 불필요할 것으로 예상되나, 확인하여 명시할 것.
6. **프로퍼티 개수**: 노트는 0개 이상 N개의 프로퍼티 값을 가질 수 있다. 전역 정의 개수와 노트당 값 개수에 상한을 둘지(예: 정의 최대 50개)와 그 근거를 기획에서 결정한다.

# 작성해야 할 항목
1. 요구사항 정의 (기능/비기능 요구사항을 구분)
2. 사용자 시나리오 / 유스케이스 (예: 정의 생성 → 노트에 값 입력 → Home에서 필터·정렬, Select 옵션 추가/삭제, 정의 이름·타입 변경, Label 마이그레이션 시나리오)
3. 입력·출력 명세 (API 엔드포인트, DTO 구조, 타입 포함 — 정의 CRUD, 노트별 값 읽기/쓰기, 필터·정렬 쿼리 파라미터. 값 쓰기가 기존 `PUT /api/notes/{id}`에 통합되는지 별도 엔드포인트인지 포함)
4. 처리 흐름 (단계별 로직, 필요 시 의사코드 — 특히 타입별 값 검증, Select 옵션 정합성 검증, 필터·정렬 쿼리 구성, 정의 삭제/타입 변경 시 기존 값 처리)
5. 인터페이스 설계 (엔티티/테이블 설계와 EF Core 마이그레이션 개요, 값 저장 모델 결정과 근거, 인덱스 전략, 클래스/메서드 시그니처 초안, 프론트 컴포넌트 구조 — 프로퍼티 패널, 정의 관리 UI, Home 필터·정렬 UI)
6. 예외 및 엣지 케이스 처리 (예: 정의 삭제 시 값 일괄 정리 — 소프트 삭제·아카이브 노트 포함 여부와 `IgnoreQueryFilters()` 검토, 사용 중인 Select 옵션 삭제, 타입 변경 시 비호환 값, 잘못된 타입의 값 입력, 낙관적 동시성 충돌과 프로퍼티 저장의 상호작용, 아카이브 노트에 대한 값 편집 시도(403), 정의 이름 중복, 계정 초기화 시 신규 테이블 정리)
7. 테스트 시나리오 (정상/경계/실패 케이스 — 서비스 로직은 `Vanadium.Note.REST.Tests`(xUnit + EF Core SQLite in-memory)에서 커버하되, PostgreSQL 전용 기능(JSONB 연산·트라이그램 등)을 쓰는 부분은 수동 검증 영역임을 명시)
8. 구현 시 고려사항 / 미결정 사항 (open questions — 예: 프로퍼티 값의 전문 검색 포함 여부, 확장 타입 로드맵(URL/Status/Relation), Label 흡수 시점, Board 뷰 연동 심화(그룹핑), 공유 노트(`/share/{token}`)에서의 프로퍼티 노출 여부)
